# Quant Trading System Design (MVP)

Date: 2026-04-17  
Scope: MVP end-to-end pipeline for A-share daily-frequency stock selection

## 1. Goals and Fixed Constraints

This design targets an engineering-first MVP with a complete runnable loop from data ingestion to backtest report.

Confirmed constraints:
- Market and frequency: A-share, daily bar.
- Universe: CSI 500.
- Prediction target: T+1 return regression.
- Strategy: Daily rebalance, long-only Top-K portfolio.
- Storage: MySQL.
- Baseline model: LightGBM.
- Backtest cost: Fixed round-trip 20 bps.
- Success criteria: End-to-end robustness and reproducibility over return threshold targets.

## 2. Architecture and Module Boundaries

The system is split into six isolated modules with explicit interfaces:

1. `data_ingest`
- Responsibility: Pull CSI 500 membership, daily bars, adjustment factors, and selected fundamentals from Tushare.
- Output: Raw ODS tables in MySQL (`ods_*`).

2. `data_pipeline`
- Responsibility: Missing value handling, suspension/abnormal filtering, trading-calendar alignment, feature joins with pandas.
- Output: Canonical sample table `dwd_features_base` keyed by `trade_date + ts_code`.

3. `feature_dataset`
- Responsibility: Convert MySQL canonical data into Qlib-compatible time-series feature dataset, perform time-window split and normalization.
- Output: Train/valid/test dataset views and versioned snapshots.

4. `model_layer`
- Responsibility: Train LightGBM regression model for T+1 return and produce daily scores.
- Output: Model registry entries and prediction scores (`ads_pred_scores`).

5. `backtest_eval`
- Responsibility: Run Top-K long-only backtest via Qlib Backtest with fixed 20 bps round-trip cost, and generate quantstats report.
- Output: NAV series, metrics, and report artifact path.

6. `orchestration`
- Responsibility: One-command pipeline execution and stage-wise rerun/retry.
- Output: Run logs and stage status records.

Design principle: modules communicate only through explicit table schemas and interfaces; no hidden cross-layer coupling.

## 3. Data Model (MySQL)

Layered schema for traceability and rerun:

1. `meta_universe`
- Columns: `trade_date, index_code, ts_code, in_out_flag`
- Purpose: Daily universe snapshot to avoid look-ahead in membership.

2. `ods_daily_bar`
- Columns: `trade_date, ts_code, open, high, low, close, vol, amount, adj_factor, is_suspended`
- Purpose: Raw daily market data landing.

3. `ods_fundamental`
- Columns: `trade_date, ts_code, pe_ttm, pb, ps_ttm, dv_ttm, ...`
- Purpose: Raw fundamentals and valuation factors.

4. `dwd_features_base` (core training sample table)
- Primary key: `trade_date + ts_code`
- Columns: Price/volume derived features (returns, volatility, momentum, volume ratios), selected fundamentals, data-quality flags such as `is_valid`.
- Constraint: Features must use information available at `trade_date` only.

5. `dws_label`
- Primary key: `trade_date + ts_code`
- Columns: `label_ret_t1`
- Rule: Generated by strict forward shift, equivalent to `close(t+1)/close(t)-1`.

6. `ads_dataset_split`
- Columns: `trade_date, split_set` (`train|valid|test`)
- Purpose: Persisted split map for reproducible experiments.

7. `ads_pred_scores`
- Columns: `run_id, trade_date, ts_code, y_pred, model_name, model_version`
- Purpose: Backtest signal source.

8. `ads_backtest_nav`, `ads_backtest_metrics`
- Purpose: Persist NAV curve and summary metrics.

9. `meta_run_log`
- Columns: `run_id, stage, status, start_time, end_time, error_msg`
- Purpose: Execution observability and failure diagnosis.

Shared requirements:
- All tables include `ingest_time` and `update_time`.
- Required indexes: `(trade_date, ts_code)` and `(run_id, trade_date)` where relevant.
- Features and labels are physically separated and joined by keys at training time for leakage auditing.

## 4. End-to-End Data Flow and Time Split

Pipeline stages:

1. `universe_snapshot`
- Fetch CSI 500 membership history and expand to trading-day snapshots into `meta_universe`.

2. `raw_ingest`
- Pull market and fundamental raw data into `ods_*` with upsert by `trade_date + ts_code`.

3. `clean_align`
- Use pandas for calendar alignment, missing-value filling, outlier clipping, and feature assembly into `dwd_features_base`.

4. `label_build`
- Build `dws_label.label_ret_t1` with per-symbol chronological `shift(-1)`.

5. `dataset_split`
- Fixed date split (configurable default):
  - train: 2016-01-01 to 2021-12-31
  - valid: 2022-01-01 to 2023-12-31
  - test: 2024-01-01 to 2025-12-31
- Persist split map in `ads_dataset_split`.

6. `fit_predict`
- Train LightGBM on train/valid and generate day-by-day predictions on test; write to `ads_pred_scores`.

7. `backtest`
- Daily rebalance long-only Top-K equal-weight portfolio with fixed round-trip 20 bps cost.

8. `evaluate_report`
- Produce Qlib metrics and quantstats HTML report, and persist outputs to `ads_backtest_*`.

Leakage prevention rules:
- Features at date `t` can only use raw data up to `t`.
- Label always uses `t+1` return.
- Normalization parameters are fit on train only, then applied to valid/test.
- Backtest reads only test-period predictions.

## 5. Error Handling and Observability

Failure handling:

1. Data ingestion failure
- Retry Tushare requests with exponential backoff.
- Record failed batch in `meta_run_log` and stop downstream stages.

2. Data quality failure
- If daily usable coverage is below threshold (default 80% of tradable CSI 500 symbols), mark day invalid and exclude from training/backtest.

3. Training failure
- Persist training context (hyperparameters, data window, sample size) and stop backtest to prevent invalid results.

4. Backtest failure
- If no eligible symbols on a day, degrade to cash-holding for that day and log warning instead of hard failure.

Observability:
- Unified `run_id` across all stages.
- Stage metrics: coverage ratio, feature missingness, label availability, training sample count, daily holdings count, turnover.
- Run artifacts: structured run summary (JSON) plus quantstats HTML path.

## 6. MVP Acceptance Criteria

Engineering closure is considered successful when:

1. Single command executes full chain from ingestion to backtest report.
2. Any stage failure can be traced to table/date/error type.
3. Successful run persists all four result categories in MySQL:
- Feature samples
- Prediction scores
- Backtest NAV
- Backtest summary metrics
4. No hard performance threshold for returns in MVP, but complete and reproducible report generation is mandatory.

## 7. Out of Scope (MVP)

- Multi-model benchmarking in production run (XGBoost/CatBoost added later).
- Minute-level data and intraday strategies.
- Market-neutral long-short portfolio.
- Production scheduler/cluster deployment.
